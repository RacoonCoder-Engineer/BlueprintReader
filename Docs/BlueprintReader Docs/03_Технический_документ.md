# Технический документ — BlueprintReader

## Архитектура модуля

BlueprintReader построен как классический трёхслойный конвейер: **UI (Slate) → `BPR_Core` (фабрика) → иерархия экстракторов**, все три слоя связаны единым контрактом данных, определённым в `BPR_Types.h`. Такое разделение ответственности намеренно простое и без лишних абстракций — плагин не пытается быть универсальным фреймворком, он решает одну задачу (превратить произвольный Blueprint-ассет в Markdown-текст для LLM) и структура кода прямо это отражает.

**UI-слой** не содержит никакой бизнес-логики извлечения данных. Его единственная забота — показать пользователю то, что вернул `BPR_Core`, и передать выбранный ассет вниз по конвейеру. Точка входа — пункт контекстного меню Content Browser, дальше — немодальные Slate-окна (`BPR_InfoWindow` для неподдерживаемых ассетов, `BPR_OutputWindow`/`SBPR_TabSwitcher` для результата извлечения); оба создаются через `FSlateApplication::AddWindow`, а не `AddModalWindow`, поэтому не блокируют остальной редактор.

**`BPR_Core`** — это plain C++ класс (`BLUEPRINTREADER_API BPR_Core`, `Public/Core/BPR_Core.h:20`), сознательно не `UObject`. Он не хранит состояние сцены и не участвует в рефлексии движка — это чистый фабричный/сервисный объект уровня модуля, инстанцируемый один раз при загрузке плагина (`BlueprintReader.cpp`, lifecycle модуля). `BPR_Core` владеет массивом экстракторов (`TArray<TSharedPtr<BPR_Extractor_Base>> Extractors`, `BPR_Core.h:47`) и предоставляет наружу четыре публичных метода:

- `RegisterAllExtractors()` — конструирует по одному экземпляру каждого из 8 конкретных экстракторов и сортирует массив по убыванию приоритета;
- `IsSupportedAsset(UObject*) const` — быстрая проверка «есть ли вообще экстрактор для этого ассета», используется UI ещё до открытия окна результата;
- `ExtractAsset(UObject*, FBPR_ExtractedData&)` — собственно извлечение;
- `GetUnsupportedAssetInfo() const` — данные для попап-окна, когда подходящего экстрактора не нашлось.

Приватный метод `FindSuitableExtractor(UObject*) const` (`BPR_Core.h:44`) инкапсулирует сам алгоритм подбора экстрактора и не протекает наружу — весь остальной код работает через два публичных метода (`IsSupportedAsset`, `ExtractAsset`), что даёт свободу менять внутренний алгоритм подбора без последствий для UI.

**Иерархия экстракторов** — это классическое дерево наследования от общей базы `BPR_Extractor_Base` (тоже plain C++, не `UObject`), с промежуточным слоем `BPR_Extractor_Object`, в который вынесена вся логика, общая для «обычных» Blueprint-классов (переменные, графы, структура компонентов). Экстракторы, чья внутренняя структура принципиально отличается от Blueprint-класса (интерфейсы, енумы, структуры, материалы), наследуются от `BPR_Extractor_Base` напрямую, минуя `BPR_Extractor_Object` — это осознанное архитектурное решение, а не недосмотр (см. явный комментарий в `BPR_Extractor_Interface.h`: "Inherits directly from BPR_Extractor_Base because interfaces have significantly different structure than regular Blueprint Classes"). Подробности иерархии и приоритетов — в отдельном разделе ниже.

Такая архитектура даёт плагину два ключевых свойства: во-первых, добавление поддержки нового типа ассета — это создание нового класса-наследника и одна строчка регистрации в `BPR_Core::RegisterAllExtractors()`, без модификации существующего кода (открыт для расширения, закрыт для изменения); во-вторых, UI полностью развязан с деталями извлечения — он работает с единственным типом данных `FBPR_ExtractedData`, независимо от того, Widget это, Actor или Material.

## Контракт данных

Контракт данных живёт в одном файле — `Public/Core/BPR_Types.h` — и является единственной точкой связи между слоем экстракторов и слоем UI. Такая централизация неслучайна: любое изменение формата вывода проходит через один заголовок, который включают оба слоя, что делает несовместимость на границе слоёв практически невозможной на этапе компиляции.

Центральная структура — **`FBPR_ExtractedData`** (`BPR_Types.h:41-48`):

```cpp
struct FBPR_ExtractedData
{
    FText Structure = FText::FromString(TEXT("No Data found"));
    FText Graph     = FText::FromString(TEXT("No Data found"));
    FText Design    = FText::FromString(TEXT("No Data found"));
    EAssetType AssetType = EAssetType::Unknown;
    FString AssetName;   // M4.2: source asset name, используется как имя файла по умолчанию при экспорте
};
```

Три текстовых поля-«вкладки» (`Structure`, `Graph`, `Design`) — уже готовый, Markdown-форматированный текст (по умолчанию в формате `EOutputFormat::Minimal`, см. ниже), который UI просто вставляет в текстовый виджет без дополнительной обработки. У всех трёх полей одинаковое значение по умолчанию — строка `"No Data found"` — это одновременно и safe-default на случай, если экстрактор забыл заполнить поле, и распознаваемый «пустой» маркер, на который завязана логика экспорта в файл (см. раздел про UI-слой): секция с таким содержимым не попадает в экспортируемый документ.

`EAssetType` (`BPR_Types.h:12-24`, `enum class : uint8`) — тег, по которому UI решает, какие вкладки показывать и как их подписывать:

```
Unknown, Blueprint, Actor, Widget, Material, MaterialFunction, ActorComponent, Enum, Structure, InterfaceBP
```

Важная деталь для тех, кто будет читать или расширять код: значение `Blueprint` присутствует в перечислении, но **на сегодняшний день ни один зарегистрированный экстрактор его не проставляет** — в системе нет универсального экстрактора для «просто Blueprint-класса без конкретного родителя» (см. Phase 3 / `BPR_Extractor_Blueprint` в разделе про иерархию). Исторически с этим же полем был связан баг: `BPR_Extractor_Interface` по ошибке проставлял `AssetType::Blueprint` вместо `InterfaceBP`, из-за чего UI не находил подходящий `case` в `switch` и откатывался на fallback-вкладку "Info" — баг исправлен, но сам факт подчёркивает, насколько тесно связаны `EAssetType` и переключатель вкладок в `SBPR_TabSwitcher`: рассинхронизация между тем, что проставляет экстрактор, и тем, что ожидает UI, — единственный класс ошибок, которого стоит остерегаться при доработке этого поля.

Поле `AssetName` (`FString`, без значения по умолчанию, то есть пустая строка) — добавлено в M4.2 и заполняется не экстрактором, а централизованно, в самом конце `BPR_Core::ExtractAsset` (`BPR_Core.cpp:100`, `OutData.AssetName = Asset->GetName()`). Такое решение снимает с каждого из 8 экстракторов обязанность помнить про это поле и гарантирует, что оно будет корректно заполнено для любого поддерживаемого типа ассета — единственная точка, где это поле не заполняется, это ранний `return` в `ExtractAsset` при `Asset == nullptr` (см. ниже).

Вспомогательная структура **`FUnsupportedAssetInfo`** (`BPR_Types.h:29-36`) обслуживает противоположный сценарий — когда ассет не поддерживается вовсе:

```cpp
struct FUnsupportedAssetInfo
{
    FText Title;
    FText MainMessage;
    FText SubMessage;
    FString GitHubURL;
    FText ButtonText;
};
```

Она возвращается из `BPR_Core::GetUnsupportedAssetInfo()` и напрямую разворачивается в параметры `BPR_InfoWindow` — то есть это, по сути, ViewModel для одного конкретного немодального окна, а не часть основного пайплайна извлечения.

Наконец, **`EOutputFormat`** (`BPR_Types.h:50-55`, `enum class : uint8`: `HumanReadable`, `Compact`, `Minimal`) — внутренний, кодовый (не UPROPERTY, не настраивается пользователем) переключатель плотности форматирования, используемый через `BPR_Extractor_Base::GetOutputFormat()`/`SetOutputFormat()` (значение по умолчанию — `Minimal`, `BPR_Extractor_Base.h:102`). Важно не путать его с пользовательской настройкой `EBPR_ExportFormat` (Markdown/PlainText) из `BPR_Settings.h` — они решают разные задачи: `EOutputFormat` управляет тем, насколько «плотно» экстрактор форматирует текст внутри `Structure`/`Graph`/`Design` (расчёт на экономию токенов LLM), а `EBPR_ExportFormat` управляет исключительно расширением и типом файла при экспорте на диск.

## Иерархия экстракторов и приоритеты

Все экстракторы реализуют единый контракт из трёх методов: `Process()`, `CanHandleAsset()`, `GetPriority()`. Дополнительно у базового класса есть метод-мост `Extract()`, который по умолчанию просто вызывает `Process()` — единственное исключение из этого правила: `BPR_Extractor_Widget`, который переопределяет `Extract()` напрямую (`BPR_Extractor_Widget.h:98`), а не полагается на мост базового класса, из-за специфики построения дерева виджетов.

Дерево наследования (все классы — plain C++, не `UObject`):

- **`BPR_Extractor_Base`** (`Public/Extractors/BPR_Extractor_Base.h:18`) — корень иерархии, общая инфраструктура логирования (`LogBlueprintReader`) и `EOutputFormat`.
  - **`BPR_Extractor_Object`** (`BPR_Extractor_Object.h:30`) — добавляет общую для всех Blueprint-классов логику: переменные, графы, структура класса.
    - **`BPR_Extractor_Actor`** — приоритет **100**
    - **`BPR_Extractor_ActorComponent`** — приоритет **90**
    - **`BPR_Extractor_Widget`** — приоритет **80**
  - **`BPR_Extractor_Interface`** — приоритет **70** (напрямую от `Base`)
  - **`BPR_Extractor_Enum`** — приоритет **50** (напрямую от `Base`)
  - **`BPR_Extractor_Structure`** — приоритет **50** (напрямую от `Base`)
  - **`BPR_Extractor_Material`** — приоритет **40** (напрямую от `Base`)
  - **`BPR_Extractor_MaterialFunction`** — приоритет **30** (напрямую от `Base`)

Регистрация всех восьми происходит в `BPR_Core::RegisterAllExtractors()` (`BPR_Core.cpp:34-63`): создаётся по одному `TSharedPtr` каждого конкретного класса, массив заполняется, затем сортируется по убыванию `GetPriority()`. Универсального «Blueprint-фолбэка» (`BPR_Extractor_Blueprint`, который подхватывал бы любой `UBlueprint`, не подошедший ни одному из специализированных экстракторов) в системе нет — эта идея зафиксирована в коде как закомментированная будущая Phase 3 и намеренно не входит в текущий скоуп ни одной завершённой вехи.

Подбор экстрактора для конкретного ассета выполняет `FindSuitableExtractor` (`BPR_Core.cpp:103-115`): метод идёт по отсортированному по приоритету массиву и возвращает первый экстрактор, чей `CanHandleAsset(Asset)` вернул `true`. Приоритет здесь работает именно как механизм разрешения неоднозначности между потенциально пересекающимися проверками, а не как произвольная метка — например, любой `AActor`-наследник и любой `UActorComponent`-наследник являются `UBlueprint`, поэтому порядок проверки (Actor раньше ActorComponent раньше Widget) критичен для корректности.

Таблица точных условий `CanHandleAsset()`:

| Экстрактор | Приоритет | Базовый класс | Условие `CanHandleAsset()` |
|---|---|---|---|
| `BPR_Extractor_Actor` | 100 | `BPR_Extractor_Object` | `Cast<UBlueprint>(Asset)` и `Blueprint->GeneratedClass->IsChildOf(AActor::StaticClass())` |
| `BPR_Extractor_ActorComponent` | 90 | `BPR_Extractor_Object` | `Cast<UBlueprint>(Asset)` и `GeneratedClass->IsChildOf(UActorComponent::StaticClass())` |
| `BPR_Extractor_Widget` | 80 | `BPR_Extractor_Object` | `Cast<UWidgetBlueprint>(Asset) != nullptr` |
| `BPR_Extractor_Interface` | 70 | `BPR_Extractor_Base` | `Cast<UBlueprint>(Asset)` и `BP->BlueprintType == BPTYPE_Interface` |
| `BPR_Extractor_Enum` | 50 | `BPR_Extractor_Base` | `Cast<UUserDefinedEnum>(Asset) != nullptr` |
| `BPR_Extractor_Structure` | 50 | `BPR_Extractor_Base` | `Asset->GetClass()->GetName() == TEXT("UserDefinedStruct")` (сравнение по имени класса, а не `Cast`) |
| `BPR_Extractor_Material` | 40 | `BPR_Extractor_Base` | `Cast<UMaterial>(Asset) != nullptr \|\| Cast<UMaterialInstance>(Asset) != nullptr` |
| `BPR_Extractor_MaterialFunction` | 30 | `BPR_Extractor_Base` | `Cast<UMaterialFunction>(Asset) != nullptr \|\| Cast<UMaterialFunctionInstance>(Asset) != nullptr` |

Отдельно стоит зафиксировать нюанс, важный для будущих доработок: **`Enum` и `Structure` делят один и тот же приоритет — 50**. Их условия `CanHandleAsset()` взаимоисключающие (проверка типа `UUserDefinedEnum` против сравнения имени класса со строкой `"UserDefinedStruct"`), поэтому на практике неоднозначного выбора не возникает. Тем не менее относительный порядок этих двух экстракторов друг относительно друга после `Extractors.Sort()` не гарантирован — `TArray::Sort` не документирован как стабильная сортировка, а компаратор трактует равные приоритеты как эквивалентные. Если в будущем появится третий экстрактор с приоритетом 50 и с пересекающимся условием `CanHandleAsset()`, это единственное место в архитектуре, где нужно будет явно развести приоритеты, чтобы поведение стало детерминированным.

`BPR_Extractor_Structure` заслуживает отдельного упоминания и по другой причине: это единственный экстрактор, определяющий тип ассета не через `Cast<T>`, а через строковое сравнение имени класса. Причина прагматичная — `UUserDefinedStruct` не всегда доступен как чистый C++-тип на всех конфигурациях сборки движка, и сравнение по имени класса оказалось более надёжным решением, чем прямой каст. Как показал реальный опыт миграции на UE 5.8 (см. раздел «Совместимость с версиями движка»), даже сам заголовочный файл этого типа менял расположение между версиями движка — что дополнительно подтверждает целесообразность отвязки логики распознавания от конкретного пути инклюда.

## UI-слой (Slate)

Точка входа в плагин — пункт контекстного меню Content Browser, добавляемый через `FBPR_ContentBrowserAssetActions::Register()` (`Private/UI/BPR_ContentBrowserAssetActions.cpp:13-27`). Регистрация расширяет секцию `"GetAssetActions"` меню `"ContentBrowser.AssetContextMenu"` и добавляет пункт с идентификатором `"ShowBPAsMD"`, видимым текстом **«Read Blueprint for AI-Assistant»** и тултипом «Convert Blueprint to Markdown via BPR Core».

Обработчик клика, `OnShowBPAsMDClicked()`, получает выбранный ассет через `ContentBrowserModule` и передаёт его в `ExecuteForObject(SelectedObject)` (`BPR_ContentBrowserAssetActions.cpp:71-134`) — это центральная функция-оркестратор всего UI-слоя, и её алгоритм линеен:

1. Спросить у `Core->IsSupportedAsset(SelectedObject)`, поддерживается ли ассет вообще.
2. Если нет — собрать `BPR_InfoWindow::FParams` из `Core->GetUnsupportedAssetInfo()` и открыть (или, если уже открыто, вывести на передний план) окно `BPR_InfoWindow`.
3. Если да — открыть `BPR_OutputWindow`, получить его `TabSwitcher`, вызвать `Core->ExtractAsset(SelectedObject, Data)` и передать результат через `TabSwitcher->SetData(Data)`.

**`BPR_InfoWindow`** (`Public/UI/BPR_InfoWindow.h`, `Private/UI/BPR_InfoWindow.cpp`) — намеренно простой plain C++ класс (не подкласс `SWindow`/`SCompoundWidget`), хранящий `TWeakPtr<SWindow>` на созданное окно. Такой подход к владению — через `TWeakPtr` плюс проверка «уже открыто → `BringToFront()`» — исключает дублирование окон при повторных кликах пользователя на неподдерживаемый ассет. Окно фиксированного размера 600×300, немаксимизируемое, несворачиваемое, topmost, и содержит: крупный жёлтый текст основного сообщения (шрифт 16), белый перенесённый по словам подзаголовок (шрифт 14), кнопку «Check for updates» (видна только если передан непустой URL — открывает ссылку через `FPlatformProcess::LaunchURL`), статичную серую строку-подсказку и кнопку «OK», закрывающую окно через `RequestDestroyWindow()`. Для сценария неподдерживаемого ассета все эти поля заполняются из `BPR_Core::GetUnsupportedAssetInfo()` — фиксированным, захардкоженным в `BPR_Core.cpp` набором текстов (Title «Blueprint Reader», MainMessage «Warning! This asset is not supported yet!» и т. д., включая ссылку на GitHub-репозиторий проекта).

**`SBPR_TabSwitcher`** — рабочая лошадка окна результата, отвечающая за построение набора вкладок под конкретный `EAssetType`. Метод `RebuildTabsFromData()` (`Private/UI/BPR_TabSwitcher.cpp:77-138`) реализует `switch` по типу ассета:

- **`Widget`** → 3 вкладки в порядке **Design, Structure, Graph** (Design — первая и вкладка по умолчанию для виджетов, что логично: для UMG-виджета визуальная раскладка обычно важнее для быстрого понимания, чем граф событий).
- **`Actor`, `ActorComponent`, `Material`, `MaterialFunction`, `InterfaceBP`** → 2 вкладки: **Structure, Graph**.
- **`Enum`, `Structure`** → 1 вкладка: **Structure** (у этих типов ассетов по определению нет ни графа выполнения, ни визуального дизайна).
- **`default`** (покрывает `Unknown` и неиспользуемое значение `Blueprint`) → 1 вкладка **Info**, построенная через `CreateErrorTabContent()`, которая выводит текст на русском языке («### Blueprint ещё не поддерживается...») вместе с числовым значением `EAssetType`. Это защитный, вторичный путь — основной UX для неподдерживаемых ассетов реализован раньше, через попап `BPR_InfoWindow`, ещё до открытия окна результата; ветка `default` в `SBPR_TabSwitcher` — это подстраховка на случай рассинхронизации между `IsSupportedAsset()` и фактическим содержимым `switch`.

Начиная с M4.2, в футере `SBPR_TabSwitcher` (нижний правый угол, `BPR_TabSwitcher.cpp:43-52`, и повторно добавляется идентично после каждой перестройки вкладок на строках 166-176, чтобы пережить `ClearTabs()`) находится постоянная кнопка **«Export to file…»** с тултипом «Save the extracted data to a .md/.txt file (format from plugin settings)». Логика экспорта детально описана в контексте настроек плагина (`UBPR_Settings`) и не дублируется здесь — с точки зрения архитектуры UI важно, что кнопка живёт вне жизненного цикла конкретного набора вкладок и переживает смену ассета.

## Совместимость с версиями движка

Стратегия совместимости с UE 5.7/5.8, принятая в M6, называется в проектной документации подходом **«A+C»**: единое дерево исходников (не отдельные ветки под каждую версию движка — вариант с раздельными ветками был рассмотрен и отклонён из-за риска расхождения кода между версиями со временем) плюс версийные guard-ы через макрос `UE_VERSION_OLDER_THAN`, централизованные в одном заголовке — `Public/Core/BPR_Compat.h`.

Правила использования этого заголовка зафиксированы прямо в его комментариях и обязательны к соблюдению при любой будущей работе над совместимостью:

- **Ни один другой файл плагина не использует `UE_VERSION_*`/`ENGINE_*_VERSION` напрямую.** Все версийные ветвления стягиваются в `BPR_Compat.h` — единственное место, где нужно искать и поддерживать подобный код.
- **Ветвление всегда «от старой версии»:** `#if UE_VERSION_OLDER_THAN(5, 8, 0) ... #else ... #endif` — обратного макроса `UE_VERSION_NEWER_OR_EQUAL_THAN` в движке не существует, поэтому условие всегда формулируется через отрицание «версия старее X».
- **Шим добавляется только реактивно** — после того как реальная ошибка компиляции на UE 5.8 доказала его необходимость. Спекулятивные guard-ы на точки кода, которые «возможно» изменятся, но пока компилируются одинаково на обеих версиях, добавлять запрещено.

На момент написания этого документа `BPR_Compat.h` содержит только объявление пространства имён `BPR::Compat`, которое **пустое** — ни одного `#if`-guard'а не потребовалось. Это не недоделанная работа, а фактический результат первой реальной попытки компиляции плагина под UE 5.8: подавляющее большинство точек кода, изначально классифицированных как «требует проверки» (в первую очередь — обход DAG выражений материалов, дублированный между `BPR_Extractor_Material` и `BPR_Extractor_MaterialFunction`, заранее помеченный как код с наибольшим риском поломки), скомпилировались без единого изменения, только с шумовыми предупреждениями движка.

Единственная реальная находка первой компиляции под 5.8 касалась не версийного guard'а, а прямой замены инклюда: заголовок `Engine/UserDefinedStruct.h`, который использовали `BPR_Extractor_Structure.h`/`.cpp`, в UE 5.8 удалён как forwarding-заголовок. Канонический путь — `StructUtils/UserDefinedStruct.h` (модуль `CoreUObject`) — существует без изменений в обеих версиях движка, поэтому исправление свелось к замене строки `#include` и не потребовало `#if`/`#else` вовсе: это «бесплатный» фикс в терминологии M6 (работает одинаково на 5.7 и 5.8, никакой ветвящейся логики не нужно).

Помимо этого единственного `#include`, в рамках задачи 6.0 было выполнено ещё два точечных исправления, тоже не требующих версийных guard-ов:

- В `BPR_Extractor_Widget.cpp` (`HandleSizeBoxProperties`) прямое чтение полей `USizeBox::bOverride_*` заменено на чтение через рефлексию (`FindFProperty<FBoolProperty>` + получение значения по имени поля), по аналогии с уже существующим паттерном `AppendBoolFlag`, использованным для ListView/TreeView в том же файле. Прямой доступ к этим полям опирался на нестабильную, помеченную как подверженную изменению деталь реализации `USizeBox`, поэтому переход на рефлексию — это устранение риска тихой поломки при будущих версиях движка, а не реакция на конкретную ошибку компиляции.
- Из `BlueprintReader.Build.cs` удалены неиспользуемые зависимости `StateTreeModule`, `GameplayStateTreeModule`, `StateTreeEditorModule` — они не использовались нигде в кодовой базе плагина и представляли риск линковки на хосте с UE 5.8, если на нём не включён плагин StateTree.

Итоговое состояние на момент завершения технической части M6: плагин собирается без единой ошибки на обеих версиях движка (5.7 и 5.8), файл `.uplugin` содержит `EngineVersion: "5.7.0"` как версию-«пол» для локальной разработки, а `Description` явно фиксирует модель поддержки: «Single-source, supports Unreal Engine 5.7 and 5.8 (EngineVersion managed at packaging time)». Для фактической публикации на Fab это дополнительно означает необходимость отдельной упаковки под каждую версию движка (Fab требует одно значение `EngineVersion` на одну «Project Version» в листинге) — это отдельный, не завершённый на момент написания документа этап (Phase 4 миграционного плана), не влияющий на структуру самого исходного кода.

## Сборка и зависимости

Сборка описана в `BlueprintReader.Build.cs` (`public class BlueprintReader : ModuleRules`). Модуль использует `PCHUsage = PCHUsageMode.UseExplicitOrSharedPCHs` и явно отключённый Unity-билд (`bUseUnity = false`) — второе осознанное решение для плагина такого размера: отключение Unity-сборки увеличивает время полной компиляции, но упрощает диагностику ошибок компиляции по отдельным файлам экстракторов на этапе активной разработки и миграции между версиями движка.

**`PublicDependencyModuleNames`** сведён к минимуму — только `["Core"]` (комментарий в коде прямо фиксирует правило: «Public is always only Core»). Это значит, что ни один внешний модуль, который потенциально мог бы зависеть от `BlueprintReader`, не унаследует транзитивно ничего, кроме `Core` — весь функциональный вес модуля инкапсулирован как приватные зависимости, что правильно для плагина-инструмента, не предоставляющего собственный публичный API для других модулей.

**`PrivateDependencyModuleNames`** (общие для всех таргетов): `Core`, `CoreUObject`, `Engine`, `UnrealEd`, `Slate`, `SlateCore`, `UMG`, `UMGEditor`, `DeveloperSettings`. Последний добавлен в M4.1 специально под `UBPR_Settings : UDeveloperSettings` — первый `UObject`-класс в модуле, из-за чего для плагина начал запускаться UnrealHeaderTool (UHT), что стоит держать в уме при добавлении новых `UCLASS`/`USTRUCT` в будущем (требования к `GENERATED_BODY()`, порядку инклюдов и т. д. становятся актуальны только с этого момента истории проекта).

Для редакторских таргетов (`if (Target.Type == TargetType.Editor)`) добавляется отдельный блок зависимостей: `UnrealEd`, `ContentBrowser`, `AssetTools`, `ToolMenus`, `BlueprintGraph`, `GraphEditor`, `KismetCompiler`, `WorkspaceMenuStructure`, `UMG`, `EditorStyle`, `DesktopPlatform`. Последний, `DesktopPlatform`, добавлен в M4.2 под нативный `SaveFileDialog`, используемый функцией экспорта в файл. Явно зафиксировано отсутствие в этом списке `StateTreeModule`/`GameplayStateTreeModule`/`StateTreeEditorModule` — они были удалены в рамках задачи M6.0(b) как неиспользуемый, рискованный для линковки на «чистом» хосте UE 5.8 груз (см. раздел о совместимости с версиями движка).

Метаданные плагина в `BlueprintReader.uplugin`: единственный модуль `"BlueprintReader"`, `Type: "Editor"` (плагин работает только в редакторе, не участвует в рантайме упакованной игры), `LoadingPhase: "Default"`, `PlatformAllowList: ["Win64", "Mac", "Linux"]`. Версия на момент написания документа — `VersionName: "0.2.0"` (`Version` (целочисленный): 2, `FileVersion`: 3), намеренно не `1.0.0` — согласно принятой в проекте семантике версионирования, переход на мажорную версию 1.0 зарезервирован за завершением вехи M5 (CI, smoke-тесты, финальный релиз на Fab), а поле `IsBetaVersion: true` отражает именно это состояние незавершённости процесса QA, а не технического ядра плагина.

## Тестирование и валидация

Формального автоматизированного набора тестов (unit-тестов на C++, automation-тестов движка) на момент написания документа в плагине нет — это открытый пункт вехи M5. Тем не менее валидация функциональности выполняется, и следует различать несколько разных, реально применённых на практике механизмов.

**Ручная редакторская валидация по вехам.** Каждая из вех M2–M4 отмечена в истории проекта как «editor-validated»: после реализации изменений плагин запускался в реальном редакторе UE, и на реальных ассетах (в том числе на вложенных `WBP_*`-виджетах с непустыми пользовательскими переменными) проверялся фактический вывод — например, валидация M2.4 подтвердила, что вложенные лейблы и конфигурационные поля (`In Text: Clan`, `MaxCharToEnter` и т. д.) действительно показывают текущие значения инстанса, а не значения по умолчанию класса. Это основной, наиболее насыщенный по факту способ проверки корректности на сегодняшний день.

**Компиляционная валидация как механизм обнаружения проблем совместимости.** В отсутствие у Epic Games публичного списка breaking changes между UE 5.7 и 5.8 для C++ API, реальная компиляция плагина под обе версии движка стала основным инструментом *обнаружения* проблем миграции, а не просто проверкой их отсутствия. Практически это означает: вместо того чтобы заранее гадать, какие участки кода могут сломаться на 5.8, и оборачивать их в спекулятивные `#if`-guard'ы, команда сначала попыталась скомпилировать существующий код как есть под 5.8 (в том числе через прямой вызов `RunUAT BuildPlugin` инструментами самого движка 5.8, без изменений в проекте-хосте — самый быстрый сигнал о компилируемости) и уже по результатам реальных ошибок компиляции точечно вносила исправления. Такой подход оказался эффективным: единственная настоящая находка — устаревший путь инклюда `Engine/UserDefinedStruct.h` — была обнаружена именно так, а не предугадана заранее.

**Ручной smoke-diff между версиями движка (задача 6.3).** Отдельным, специально проведённым шагом стала прогонка одних и тех же реальных ассетов через плагин, собранный под UE 5.7 и под UE 5.8, с ручным сравнением вывода бок о бок. Это критически важная проверка именно потому, что она ловит класс ошибок, недоступный компилятору: тихие переименования рефлексируемых полей движка (доступ по имени свойства, как в `BPR_Extractor_Structure` при сравнении имени класса, или потенциально в местах, читающих свойства виджетов по строковому имени) не вызывают ошибку компиляции — они просто молча начинают возвращать пустое значение или `nullptr` в рантайме. Диффовая проверка вывода между версиями — единственный практический способ отловить такой класс регрессии на плагине этого профиля. Результат проверки — вывод совпадает между версиями движка.

**Открытые задачи по вехе M5** (пока не сделаны, зафиксированы как явный технический долг процесса, а не кода): автоматизация compile-матрицы 5.7+5.8 в CI (сейчас проверка компиляции выполняется вручную, локально), формальные smoke-тесты сверх уже проведённой вручную задачи 6.3, и сама подготовка релиза на Fab (упаковка под каждую версию движка отдельно, публикация Project Version).

## Известные ограничения и технический долг

Ниже — открытые пункты, зафиксированные по факту как ещё не сделанные, без попытки представить их лучше или хуже, чем они есть.

- **Нет автоматизированного тестирования.** Как описано в разделе о валидации, вся текущая проверка корректности — ручная (редакторская валидация по вехам, ручной smoke-diff между версиями движка). CI compile-матрица 5.7+5.8 и формальные автоматизированные smoke-тесты — открытая часть вехи M5, не начатая как отдельный процесс на момент написания документа.
- **Нет универсального fallback-экстрактора для произвольных Blueprint-классов.** Ассеты типа `UBlueprint`, не являющиеся ни `AActor`-, ни `UActorComponent`-наследником и не интерфейсом (например, произвольный служебный Blueprint-класс без конкретного родителя, а также `UAnimBlueprint` — он технически является `UBlueprint`-наследником, но не подпадает ни под одно текущее условие `CanHandleAsset()`), считаются неподдерживаемыми и обрабатываются через попап `BPR_InfoWindow`, а не через содержательное извлечение. Идея генерического `BPR_Extractor_Blueprint` зафиксирована в кодовой базе как закомментированная будущая Phase 3, статус — намеренно отложено, не в скоупе ни одной завершённой вехи.
- **Обход DAG выражений материалов не переиспользуется между `BPR_Extractor_Material` и `BPR_Extractor_MaterialFunction`.** Логика дублирована между двумя экстракторами по историческим причинам и была изначально помечена как код с наибольшим риском при миграции на новую версию движка. На практике при переходе на UE 5.8 этот код скомпилировался без единого изменения, но дублирование само по себе остаётся техническим долгом: любое будущее исправление или улучшение логики обхода графа выражений придётся вносить в двух местах синхронно, если не провести отдельный рефакторинг по вынесению общей части.
- **`CleanName`: остаточное дублирование логики нормализации имён.** После завершения вех M2/M3 в базовых классах эта логика была унифицована почти везде; единственное известное оставшееся дублирование — локальная для виджетов копия в `BPR_Extractor_Widget`, которая пока не сведена к общей реализации из базового класса.
- **Логирование не полностью унифицировано.** Часть неWidget-файлов (UI-слой, `BPR_Extractor_Material`, `BPR_Extractor_ActorComponent`) всё ещё использует стандартную категорию логирования движка `LogTemp` вместо собственной категории плагина `LogBlueprintReader`, определённой в `BPR_Types.h`. Это не влияет на функциональность, но снижает удобство фильтрации логов плагина в Output Log при диагностике проблем пользователями и разработчиками.
- **Качество Design-вывода для изображений и цвета.** В выводе вкладки Design ссылки на визуальные ресурсы (`ResourceObject` у `Brush`/`Image`) в текущей реализации выводятся как сырой GUID объекта, а не как читаемое имя ассета — для LLM-потребителя это существенно менее полезная информация, чем имя текстуры/материала. Аналогично, полностью нулевые/прозрачные значения цвета (например, `BackgroundColor:(0,0,0,0)`) сейчас не отфильтровываются и засоряют вывод неинформативными строками. Оба пункта зафиксированы как открытый пункт качества вывода вехи M4, не блокирующий функциональность, но снижающий полезность вывода для конечного сценария использования (копирование в чат с ИИ-ассистентом).
- **Отложенные пункты UX (M4.3, M4.4).** Фильтр частичного графа (возможность извлекать не весь граф целиком, а выбранную часть) и индикатор прогресса извлечения для крупных ассетов были осознанно вынесены за пределы скоупа M4 и остаются в бэклоге без привязки к конкретной вехе.
- **Публикация на Fab требует отдельного шага упаковки под каждую версию движка.** Поскольку поле `EngineVersion` в `.uplugin`-файле может содержать только одно значение, а не список поддерживаемых версий, для полноценной публикации на Fab потребуется дважды упаковать плагин — отдельно с `EngineVersion: "5.7.0"` и отдельно с `"5.8.0"` — как отдельные Project Version в рамках одного листинга. Сам исходный код при этом не меняется; это чисто процессный, не кодовый технический долг, отнесённый к Phase 4 миграционного плана и вехе M5.
