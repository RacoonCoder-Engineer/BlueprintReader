# Design Document — BlueprintReader

## Functionality Overview

**BlueprintReader** is an Unreal Engine (5.7 and 5.8) editor plugin that turns any supported Blueprint-family asset into clean, structured Markdown text, ready to paste into a conversation with an AI assistant (Claude, ChatGPT, Gemini, etc.). The plugin's idea is pragmatic in the extreme: the developer opens the Content Browser, right-clicks an asset, picks a menu entry — and within a fraction of a second gets a window with text that can be copied and pasted into an LLM chat to ask it to explain the logic, find a bug, suggest a refactor, or generate a C++ equivalent.

The key architectural principle that shapes the entire user experience: the plugin **never calls any external API and never stores any keys**. All the work is done locally, powered by Unreal Engine's reflection system — the plugin reads the graph, variables, components, and widgets through the standard UObject/UEdGraph API and formats the result as text. What to do with that text afterward (which chatbot to use, copy to clipboard, save to a file, add to an agent's system prompt) is entirely up to the user. This is a deliberate rejection of the "one button — instant AI answer" model: BlueprintReader is a context-preparation tool, not a code generator.

Functionally, the plugin addresses three overlapping needs:

- **Fast documentation.** Instead of manually describing in words what a Blueprint does, the developer gets an accurate textual snapshot of its structure in seconds — useful both for project history and for onboarding new team members.
- **Preparing context for an AI assistant.** Modern LLMs analyze text excellently and are noticeably worse at reading screenshots of node graphs. BlueprintReader closes exactly that gap: it turns a visual graph into text an LLM parses natively.
- **Diagnostics and refactoring.** The structured output (variables, graphs, event bindings, widget hierarchy) helps spot illogical dependencies, duplicated logic, or forgotten stubs faster on its own — even without any AI involvement.

The product is distributed as MIT-licensed open source and simultaneously listed on Fab under a dual model: **free for personal use, and 10 USD for studio/business use**. This combination — open source plus a lenient commercial model — is itself part of the value proposition compared to typical paid marketplace plugins (see `01_Concept_and_Vision.md`, section "Difference from Existing Solutions").

## User Interface

The entire user experience is built on Slate (the editor's native UI framework), with no third-party UI dependencies, and boils down to a single "right-click → menu item" action plus up to two kinds of popup windows.

**Entry point.** The plugin adds a menu entry to the Content Browser's standard context menu (`GetAssetActions` section) with the identifier `ShowBPAsMD`, visible label — **"Read Blueprint for AI-Assistant"**, tooltip — "Convert Blueprint to Markdown via BPR Core". The entry appears on right-click for any selected asset — the plugin does not pre-filter the list by type; instead it checks support at click time and adapts its response (see below). This is a deliberate decision: the menu entry is always available and predictable, rather than appearing and disappearing depending on asset type, which would be less obvious to the user.

**The "supported / not supported" fork.** After the click, the plugin uses `BPR_Core::IsSupportedAsset()` to determine whether any of the registered extractors is able to handle the selected asset:

- **If the asset is not supported** — a compact information window opens (`BPR_InfoWindow`): non-modal, fixed size 600×300, cannot be maximized or minimized, always on top of other windows. It displays, in large yellow text, the message "Warning! This asset is not supported yet!", with white explanatory text below stating that this is not an engine error but simply support for this asset type not yet being implemented, a "Check for updates" button (leads to the plugin's GitHub repository), and a static gray hint about the roadmap. An "OK" button closes the window. If the window is already open, clicking again simply brings it to the front instead of spawning a duplicate.
- **If the asset is supported** — the main output window opens (`BPR_OutputWindow`), and the `TabSwitcher` placed inside it is populated with the data just extracted by `BPR_Core::ExtractAsset()`.

**Main window and tab switcher.** Inside `BPR_OutputWindow` lives `SBPR_TabSwitcher` — a widget that rebuilds its set of tabs based on the asset type (`EAssetType`):

- **Widget assets** (`EAssetType::Widget`) get three tabs in the order **Design → Structure → Graph**, with Design opening first by default — for UMG widgets, the visual layout is usually of more immediate interest than the logic graph.
- **Actor, ActorComponent, Material, MaterialFunction, InterfaceBP** get two tabs: **Structure → Graph** — these asset types have no separate "visual" representation, so Design is not shown.
- **Enum and Structure** get only one tab — **Structure**, since enumerations and data structures have neither an executable graph nor a visual layout.
- Any other case (primarily `Unknown`, as well as the technically existing but never-assigned value `Blueprint`) falls back to an **Info** tab with a Russian-localized placeholder message along the lines of "Blueprint is not yet supported...", including the numeric value of `EAssetType`. This path is a secondary safety mechanism for unexpected data states; under normal operation the user should never see it, because the primary path for unsupported assets is the `BPR_InfoWindow` described above, shown before the output window is ever opened.

**Persistent footer with an export button.** Regardless of how many tabs are shown or how often they are rebuilt, the bottom-right corner of the `TabSwitcher` always contains an **"Export to file…"** button with the tooltip "Save the extracted data to a .md/.txt file (format from plugin settings)". The button is implemented so that it survives a full rebuild of the tab set (`ClearTabs()` + re-construction) — meaning the user never loses access to export while switching between different assets within an editor session.

Thus, the whole interface is deliberately minimalist: one context menu, one placeholder information window for unsupported types, and one tabbed output window plus an export button — no wizards, no on-the-fly configuration dialogs, no multi-step flows.

## Supported Asset Types

Support for an asset type is not determined by a hard-coded list of paths/classes in the UI, but by whether at least one registered extractor's `CanHandleAsset()` method returns `true` for the given object. At present, eight extractors are registered, and accordingly the following types are supported:

1. **Actor Blueprints** — any `UBlueprint` whose `GeneratedClass` derives from `AActor`.
2. **ActorComponent Blueprints** — any `UBlueprint` whose `GeneratedClass` derives from `UActorComponent`.
3. **Widget Blueprints (UMG)** — any `UWidgetBlueprint`, including both regular screen widgets and nested custom widgets (`WBP_*`, `UUserWidget`).
4. **Blueprint Interfaces (BPI)** — any `UBlueprint` with `BlueprintType == BPTYPE_Interface`.
5. **User-Defined Enums** — any `UUserDefinedEnum`.
6. **User-Defined Structures** — assets whose dynamic class name matches the string `"UserDefinedStruct"`.
7. **Materials** — `UMaterial` and `UMaterialInstance`.
8. **Material Functions** — `UMaterialFunction` and `UMaterialFunctionInstance`.

Extractor priorities (100 → 30, higher meaning more specific) are used as the check order when looking for a suitable handler, not as a way to resolve conflicts between several simultaneously matching extractors — in practice the `CanHandleAsset()` conditions are mutually exclusive, so no ambiguous choice arises.

**What is not supported in the current version.** Any asset that does not fall into one of the eight categories above is considered unsupported and receives the placeholder information window instead of the output window. In particular, it is important to clearly understand that a "Blueprint" in the everyday sense is not the same thing as "a Blueprint that BlueprintReader knows how to extract": for example, an **Animation Blueprint** (`UAnimBlueprint`) is technically a `UBlueprint`-derived class, but is neither an `AActor`, nor a `UActorComponent`, nor an interface — and is therefore **not supported**, despite looking like "just another Blueprint" to the user. Likewise unsupported: ordinary non-Actor/non-Component Blueprint classes without a suitable parent, sound assets, textures, Data Assets and Data Tables, Niagara systems, level/world assets, StateTree/Behavior Tree, and other engine data types. This limitation is stated plainly and honestly in the "Limitations of the Current Version" section below, and is one of the main vectors for future product expansion.

## Output Format (Structure / Graph / Design)

The extraction result is not a single block of text but three independent, thematically separated fields, represented in code by the `FBPR_ExtractedData` structure (`Structure`, `Graph`, `Design` — all of type `FText`, plus `AssetType` and `AssetName`). The split into three fields is driven both by ease of reading for humans and by the AI assistant's convenience: an LLM works far more effectively with text broken into thematic blocks than with a single unbroken wall of text.

- **Structure** — a "static" slice of the Blueprint: variables (with types and, where applicable, default values), functions and their signatures, components (for Actor/ActorComponent), implemented interfaces, exposed properties, and so on. This answers the question "what is this made of."
- **Graph** — a "dynamic" slice: the contents of event and function graphs, node chains, connections via exec pins and via data, bindings of handlers to events (including, for widgets, `K2Node_ComponentBoundEvent`-style bindings between a visual element and its handler in code). This answers the question "what does this do."
- **Design** — a widget-specific slice: the UMG widget hierarchy, their mutual nesting, key visual and layout properties (for example, `SizeBox` overrides, visibility, text bindings, etc.). Populated only for asset types that have a visual component; for other types the field remains equal to the sentinel value "N/A" or "No Data found" (see below).

Each of the three fields is by default formatted as Markdown (headings, lists, emphasis) according to the internal `EOutputFormat` flag (default value — `Minimal`, the most compact option, optimized for saving tokens when pasted into an LLM chat) — this is a separate, internal level of code formatting, not to be confused with the user-facing setting for the exported file's format (`EBPR_ExportFormat`, see the "Plugin Settings" section below), which concerns the file extension the result is saved with on disk.

The meaning of "empty" values deserves special emphasis. If a given asset type logically has no data for a particular tab (for example, Design for an Actor Blueprint, or Graph for a User-Defined Structure), the field is not left as a physically empty string but instead receives one of two predictable placeholder values — `"N/A"` or `"No Data found"`. This is not a cosmetic detail: it is precisely by these two sentinel strings that the export logic (`IsMeaningful()`, see below) distinguishes "there really is nothing to show here" from "there is content here," and excludes empty sections from the final exported document.

Finally, every extraction result carries an accompanying `AssetName` (the source asset's name) — it is not shown as a separate "tab," but is used as the default file name on export, ensuring that a saved file is immediately recognizable by name even if the user extracted data for several different Blueprints in a row.

## Plugin Settings

Settings live in the standard editor section **Project Settings → Plugins → Blueprint Reader** (class `UBPR_Settings`, a subclass of `UDeveloperSettings`, editor-level configuration — `config = Editor`). Choosing `UDeveloperSettings` is a deliberate architectural decision: it is the standard mechanism familiar to any UE developer for configuring a plugin, requires no separate settings window inside BlueprintReader itself, and is automatically integrated into Project Settings search.

Three settings are available, grouped into two categories:

**"Output" category:**
- **Export Format** (`ExportFormat`, enum `EBPR_ExportFormat`) — determines the file format used when exporting to disk. Two options: **Markdown (.md)** (the default value) and **Plain Text (.txt)**. This setting directly controls the file extension offered by the save dialog's filter, and, correspondingly, the expectation of whether the saved text will contain Markdown markup or be plain unformatted text.

**"Widget" category:**
- **Restrict Widget Recursion Depth** (`bRestrictWidgetRecursionDepth`, boolean, default `true`) — enables a limit on the traversal depth of the nested widget tree during extraction. Widgets in UMG are frequently nested many levels deep (panel → container → sub-widget → sub-sub-widget...), and without a depth limit, traversing the recursive tree on a heavy UI asset risks generating an excessively long output, poorly suited for pasting into an LLM chat (context limits, loss of readability).
- **Widget Recursion Depth** (`WidgetRecursionDepth`, integer, range 1–50, default 10) — the actual depth value the recursive traversal (`ProcessWidgetHierarchy`) goes down to. The field is active and editable only when `bRestrictWidgetRecursionDepth` is enabled (`EditCondition` in the UPROPERTY) — that is, it is visually locked in the settings UI when the restriction is turned off, so as not to create the false impression that the value has any effect on its own.

From a user-scenario standpoint, these two settings exist primarily to balance extraction completeness against practicality of the result: users working with very deeply nested custom widgets (for example, complex HUDs with many reusable `WBP_*` components) can either increase the depth to get a fuller picture, or disable the restriction entirely if completeness matters more than compactness. The default value (10 levels, restriction enabled) was chosen as a reasonable middle ground that covers the overwhelming majority of real-world UI hierarchies without risking a runaway, excessively long output.

All three settings are read from `UBPR_Settings` via `GetDefault<UBPR_Settings>()` at the moment of extraction (for recursion depth — inside `BPR_Extractor_Widget`) and at the moment of export (for file format — inside `SBPR_TabSwitcher`), meaning a settings change in Project Settings applies to the next extraction/export immediately, with no need to restart the editor.

## Data Export

The **"Export to file…"** button in the output window's footer turns the already-extracted data (`Structure`, `Graph`, `Design` from the current tab session) into a single text document and saves it to disk via the operating system's native save-file dialog.

**Document assembly.** The final file is a concatenation of the three fields in a fixed order, **Structure → Graph → Design**, separated by blank lines, with one important caveat: a section is included in the document only if it is meaningful. The meaningfulness check (an internal predicate function) trims whitespace from the edges of the text and discards the section if, after trimming, it turns out to be empty or equal to one of the two sentinel values — `"N/A"` or `"No Data found"`. In other words, if a user exports, say, a User-Defined Structure (which has only Structure, with Graph and Design being placeholders), the file will contain only the Structure content, without the two placeholder lines cluttering the document. The final text is additionally trimmed of trailing whitespace characters.

**The "nothing to export" scenario.** If, after assembly, the document turns out to be completely empty (i.e., all three sections were deemed non-meaningful), the button does not open the save dialog at all, and instead immediately shows a **"Nothing to export for this asset."** notification with a failure icon — the user doesn't waste time choosing a save path for an empty file.

**Format selection and save dialog.** The format is taken from the `ExportFormat` setting (see above): `Markdown` yields the `.md` extension and the filter "Markdown Document (*.md)|*.md"; any other value (or the absence of a settings object) yields the `.txt` extension and the filter "Text Document (*.txt)|*.txt" (that is, Markdown is the default behavior, including in the absence of explicitly configured settings). The default file name is formed as `{AssetName of the current data, or "BlueprintExport" if not set}.{extension}`. The save dialog is a native, platform `IDesktopPlatform::SaveFileDialog`, opened in the project's `FPaths::ProjectSavedDir()` directory, with the title "Export Blueprint Data"; the parent window for the dialog is determined via `FSlateApplication::FindBestParentWindowHandleForDialogs`, so that the dialog is modally attached to the editor window rather than getting "lost" behind other windows.

**Cancellation and result.** If the user closes the dialog without choosing a file, the plugin silently ends the operation, with no notifications of any kind (this is expected, unobtrusive behavior: canceling a save is not an error). If a file is chosen, the document is written via `FFileHelper::SaveStringToFile` in **UTF-8 without BOM** encoding — an important detail for correctly displaying Cyrillic and any non-ASCII characters (for example, Russian-language comments in a Blueprint) in most text editors and Markdown viewers. The user learns whether the write succeeded or failed from a popup editor notification (`FSlateNotificationManager`) with a 4-second lifetime and a corresponding success/failure icon: **"Exported to {file name}"** on success (plus a log entry), or **"Export failed (could not write file)."** on failure (plus a warning in the log) — for example, if the target path turned out to be unwritable.

Thus, export is designed to be a maximally "quiet," predictable operation: minimal dialogs, clear feedback only when it is truly needed (success/failure of the write, absence of data), and no noise on ordinary cancellation.

## User Scenarios (Usage Flows)

**Scenario 1 — "Explain this Blueprint."** A developer inherits someone else's (or their own long-untouched) Actor Blueprint and doesn't understand what's happening in its event graph. They right-click the asset → "Read Blueprint for AI-Assistant" → in the window that opens, read the Structure tab (which variables and components exist) and the Graph tab (what logic connects the events), press "Export to file…" if needed, save the `.md` file, and paste its contents into a chat with an AI assistant asking it to "explain what this Blueprint does." Because the output is already structured into sections and formatted as Markdown, the LLM doesn't waste context parsing visual noise and gives a substantive answer right away.

**Scenario 2 — "Help with a bug in a widget."** A UI programmer notices that a nested `WBP_*` widget on screen is displaying the wrong value. They open the widget via BlueprintReader, land straight on the **Design** tab (widgets open on it by default), see the nesting hierarchy and the exposed properties of the specific instance (including non-default values set on that particular instance — for example, `In Text: Clan`), then switch to the **Graph** tab to check the event-to-handler bindings. The combination of Design + Graph gives the AI assistant the full picture of both "what is visible" and "what executes" — without the user having to describe the structure in words themselves.

**Scenario 3 — "The asset is not supported."** A user mistakenly (or out of curiosity) tries to read, say, an Animation Blueprint or a Data Table. After clicking the menu item, instead of a window with data, a compact warning window appears with a clear message that this asset type is not yet supported, plus a link to the GitHub repository to check for updates. The user does not get an empty or broken output — the message is unambiguous and leaves no impression that something is malfunctioning.

**Scenario 4 — "Quick documentation for the team."** A team lead, ahead of a code review or onboarding a new hire, wants a textual description of the project's key Blueprint classes. They run several Actor and Component Blueprints through BlueprintReader one after another, pressing "Export to file…" each time and saving the result into the project's documentation folder (`Saved/` or a separate docs folder) — without having to manually type out a description of the structure, the developer gets a text snapshot that is accurate as of the moment of export.

**Scenario 5 — "Adjusting the depth for a complex UI."** A UI designer working on a heavy HUD with many nested, reusable widgets notices that BlueprintReader's output for the root screen cuts off earlier than expected (the default depth is 10 levels). They go to Project Settings → Plugins → Blueprint Reader, increase `Widget Recursion Depth` or disable `Restrict Widget Recursion Depth` entirely, and reopen the same asset via the context menu — the new setting value applies immediately, with no editor restart required.

## Limitations of the Current Version

The current version of BlueprintReader (0.2.0, beta status — `IsBetaVersion: true`) is deliberately limited in the scope of asset types it covers and in the depth of certain scenarios. Below is the full list of known limitations the user should understand before starting to work with it:

- **Limited set of supported types.** Only eight asset categories are supported (Actor, ActorComponent, and Widget Blueprints, Blueprint interfaces, user-defined enums and structures, materials and material functions — see the "Supported Asset Types" section). In particular, **Animation Blueprints are not supported**, despite formally being a `UBlueprint`-derived class — this is a frequent source of confusion for new users. Also unsupported: StateTree/Behavior Tree assets, Niagara systems, Data Assets/Data Tables, and any ordinary Blueprint classes that are neither Actor, nor ActorComponent, nor an interface.
- **No subgraph filtering.** The plugin extracts the graph in its entirety; selective export of "just this function" or "just this part of the graph" (a planned partial-graph-filtering capability) is absent in the current version and deferred to the future.
- **No progress indicator.** Data extraction for large, content-heavy assets runs synchronously, with no visual progress indicator; in practice the operation is fast enough for the overwhelming majority of assets, but the UI provides no feedback about what exactly is happening while processing especially heavy graphs.
- **Image/brush resolution in the Design output.** For widgets that use a `Brush`/`Image` referencing an external resource (`ResourceObject`), the Design tab in the current version outputs the resource's raw identifier (GUID) instead of a human-readable asset name — a known output-quality improvement item, not yet implemented.
- **Non-meaningful color values are not filtered out.** The Design output may contain visually useless lines such as `BackgroundColor:(0,0,0,0)` (a fully transparent/zero color) — filtering out such non-meaningful values is planned but not yet implemented.
- **A single source-code branch for two engine versions (5.7 and 5.8).** The plugin builds and has been verified on both engine versions from a single source tree, with no version-specific `#if` branches (the `BPR_Compat.h` file is reserved for future targeted fixes, but is empty as of this document's writing). The risk is that future engine versions could potentially require version-specific changes — this mechanism is prepared, but there are no concrete workarounds yet, since none have been needed.
- **No automated CI and no formal smoke tests.** Compatibility with 5.7/5.8 has been confirmed via manual builds and manual comparison of output on real assets, but an automated compilation pipeline (a CI compile matrix) and formalized smoke tests have not yet been set up — this is a separate, not-yet-started phase of work (the product's next milestone).
- **Publishing to Fab is not yet complete for both engine versions.** Since a single `.uplugin` cannot simultaneously declare support for multiple engine versions for Fab publishing (separate "Project Versions" with a stamped `EngineVersion` are required for each), the process of preparing separate packages for 5.7 and 5.8 for the marketplace has not yet been carried out as of this document's writing.
- **Beta status.** The plugin is marked `IsBetaVersion: true` — this reflects the fact that the quality-control stage ahead of a stable release (automated testing, CI, full publication on Fab) has not yet been completed, and, per SemVer convention, the version has deliberately not been bumped to 1.0.0.
