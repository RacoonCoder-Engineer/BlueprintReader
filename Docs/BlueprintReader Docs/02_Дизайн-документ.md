# Дизайн-документ — BlueprintReader

## Обзор функциональности

**BlueprintReader** — это редакторский плагин Unreal Engine (5.7 и 5.8), который превращает любой поддерживаемый ассет Blueprint-семейства в чистый, структурированный Markdown-текст, готовый для вставки в диалог с AI-ассистентом (Claude, ChatGPT, Gemini и т.п.). Идея плагина предельно прагматична: разработчик открывает Content Browser, кликает правой кнопкой по ассету, выбирает пункт меню — и через долю секунды получает окно с текстом, который можно скопировать и вставить в чат с LLM, чтобы попросить объяснить логику, найти баг, предложить рефакторинг или сгенерировать C++-аналог.

Ключевой архитектурный принцип, определяющий весь пользовательский опыт: плагин **не звонит ни в один внешний API и не хранит никаких ключей**. Вся работа выполняется локально, силами reflection-системы Unreal Engine — плагин читает граф, переменные, компоненты и виджеты через штатные UObject/UEdGraph API и форматирует результат в текст. Что делать с этим текстом дальше (какой чат-бот использовать, копировать в буфер, сохранять в файл, добавлять в системный промпт агента) — полностью решает пользователь. Это осознанный отказ от модели «одна кнопка — готовый ответ от AI»: BlueprintReader — инструмент подготовки контекста, а не генератор кода.

Функционально плагин решает три пересекающиеся задачи:

- **Быстрое документирование.** Вместо того чтобы вручную описывать словами, что делает Blueprint, разработчик получает точный текстовый снимок его структуры за секунды — полезно как для истории проекта, так и для онбординга новых членов команды.
- **Подготовка контекста для AI-ассистента.** Современные LLM прекрасно анализируют текст и заметно хуже — скриншоты графов нод. BlueprintReader закрывает именно этот разрыв: превращает визуальный граф в текст, который LLM разбирает нативно.
- **Диагностика и рефакторинг.** Структурированный вывод (переменные, графы, привязки событий, иерархия виджетов) сам по себе помогает быстрее находить нелогичные зависимости, дублирующуюся логику или забытые заглушки — даже без участия AI.

Продукт распространяется как MIT-лицензированный опенсорс и одновременно размещается на Fab с двойной моделью: **бесплатно для личного использования и 10 USD для использования в студии/бизнесе**. Эта комбинация — открытый исходный код плюс щадящая коммерческая модель — сама по себе является частью ценностного предложения на фоне типичных платных плагинов маркетплейса (см. `01_Концепция_и_видение.md`, раздел «Отличие от существующих решений»).

## Пользовательский интерфейс

Весь пользовательский опыт построен на Slate (родной UI-фреймворк редактора Unreal), без сторонних UI-зависимостей, и укладывается в одно действие «правый клик → пункт меню» плюс два возможных типа всплывающих окон.

**Точка входа.** Плагин добавляет пункт меню в стандартное контекстное меню Content Browser (секция `GetAssetActions`) с идентификатором `ShowBPAsMD`, видимый текст — **«Read Blueprint for AI-Assistant»**, подсказка при наведении — «Convert Blueprint to Markdown via BPR Core». Пункт появляется при клике правой кнопкой по любому выбранному ассету — плагин не фильтрует список заранее по типу, а вместо этого проверяет поддержку в момент клика и адаптирует реакцию (см. ниже). Это осознанное решение: пункт меню всегда доступен и предсказуем, а не то появляется, то исчезает в зависимости от типа ассета, что было бы менее очевидно пользователю.

**Развилка «поддерживается / не поддерживается».** После клика плагин через `BPR_Core::IsSupportedAsset()` определяет, есть ли среди зарегистрированных экстракторов хотя бы один, способный обработать выбранный ассет:

- **Если ассет не поддерживается** — открывается компактное информационное окно (`BPR_InfoWindow`): немодальное, фиксированного размера 600×300, без возможности развернуть или свернуть, всегда поверх других окон. В нём крупным жёлтым текстом выводится сообщение «Warning! This asset is not supported yet!», ниже — белый поясняющий текст о том, что это не ошибка движка, а просто ещё не реализованная поддержка данного типа ассета, кнопка «Check for updates» (ведёт на GitHub-репозиторий плагина) и статичная серая подсказка про дорожную карту. Кнопка «OK» закрывает окно. Если окно уже открыто, повторный клик просто выводит его на передний план, не плодя дубликаты.
- **Если ассет поддерживается** — открывается основное окно вывода (`BPR_OutputWindow`), а вставленный в него `TabSwitcher` заполняется данными, только что извлечёнными `BPR_Core::ExtractAsset()`.

**Основное окно и переключатель вкладок.** Внутри `BPR_OutputWindow` живёт `SBPR_TabSwitcher` — виджет, который на основе типа ассета (`EAssetType`) перестраивает набор вкладок:

- **Widget-ассеты** (`EAssetType::Widget`) получают три вкладки в порядке **Design → Structure → Graph**, причём Design открывается первой по умолчанию — для UMG-виджетов визуальная раскладка обычно интереснее в первую очередь, чем граф логики.
- **Actor, ActorComponent, Material, MaterialFunction, InterfaceBP** получают две вкладки: **Structure → Graph** — для этих типов ассетов нет отдельного «визуального» представления, поэтому Design не показывается.
- **Enum и Structure** получают только одну вкладку — **Structure**, поскольку у перечислений и структур данных нет ни исполняемого графа, ни визуального макета.
- Любой прочий случай (в первую очередь `Unknown`, а также технически существующее, но нигде не присваиваемое значение `Blueprint`) даёт запасную вкладку **Info** с локализованным на русском сообщением-заглушкой вида «Blueprint ещё не поддерживается...», включающим числовое значение `EAssetType`. Этот путь — вторичный защитный механизм на случай непредвиденного состояния данных; в штатном сценарии пользователь вообще не должен его увидеть, потому что основной путь для неподдерживаемых ассетов — это уже описанное выше окно `BPR_InfoWindow`, показываемое ещё до открытия окна вывода.

**Постоянный футер с кнопкой экспорта.** Независимо от того, сколько вкладок показано и как часто они перестраиваются, в правом нижнем углу `TabSwitcher` всегда присутствует кнопка **«Export to file…»** с подсказкой «Save the extracted data to a .md/.txt file (format from plugin settings)». Кнопка реализована так, что переживает полную перестройку набора вкладок (`ClearTabs()` + повторное построение) — то есть пользователь никогда не теряет доступ к экспорту при переключении между разными ассетами в рамках сессии редактора.

Таким образом, весь интерфейс сознательно минималистичен: одно контекстное меню, одно информационное окно-заглушка для неподдерживаемых типов и одно окно вывода с вкладками плюс кнопка экспорта — никаких мастеров, диалогов настройки «на лету» или множественных этапов.

## Поддерживаемые типы ассетов

Поддержка типа ассета определяется не жёстким списком путей/классов в UI, а тем, находится ли среди зарегистрированных экстракторов хотя бы один, чей метод `CanHandleAsset()` возвращает `true` для конкретного объекта. На данный момент зарегистрировано восемь экстракторов, и, соответственно, поддерживаются следующие типы:

1. **Actor Blueprints** — любой `UBlueprint`, чей `GeneratedClass` наследуется от `AActor`.
2. **ActorComponent Blueprints** — любой `UBlueprint`, чей `GeneratedClass` наследуется от `UActorComponent`.
3. **Widget Blueprints (UMG)** — любой `UWidgetBlueprint`, включая как обычные виджеты экрана, так и вложенные пользовательские виджеты (`WBP_*`, `UUserWidget`).
4. **Blueprint Interfaces (BPI)** — любой `UBlueprint` с `BlueprintType == BPTYPE_Interface`.
5. **User-Defined Enums** — любой `UUserDefinedEnum`.
6. **User-Defined Structures** — ассеты, чьё динамическое имя класса совпадает со строкой `"UserDefinedStruct"`.
7. **Materials** — `UMaterial` и `UMaterialInstance`.
8. **Material Functions** — `UMaterialFunction` и `UMaterialFunctionInstance`.

Приоритеты экстракторов (100 → 30, где выше — специфичнее) используются как порядок проверки при поиске подходящего обработчика, а не как способ разрешения конфликтов между несколькими одновременно подходящими экстракторами — на практике условия `CanHandleAsset()` взаимоисключающие, поэтому неоднозначного выбора не возникает.

**Что не поддерживается в текущей версии.** Любой ассет, не попадающий ни в одну из восьми категорий выше, считается неподдерживаемым и вместо окна вывода получает информационное окно-заглушку. В частности, важно явно понимать, что «Blueprint» в бытовом смысле — не то же самое, что «Blueprint, который умеет извлечь BlueprintReader»: например, **Animation Blueprint** (`UAnimBlueprint`) технически является `UBlueprint`-производным классом, но не является ни `AActor`, ни `UActorComponent`, ни интерфейсом — и поэтому **не поддерживается**, несмотря на то, что выглядит как «ещё один Blueprint» для пользователя. Аналогично не поддерживаются: обычные не-Actor/не-Component Blueprint-классы без подходящего родителя, звуковые ассеты, текстуры, Data Assets и Data Tables, Niagara-системы, ассеты уровней/миров, StateTree/Behavior Tree и другие типы данных движка. Это ограничение прямо и честно описано в разделе «Ограничения текущей версии» ниже и является одним из главных векторов будущего расширения продукта.

## Формат вывода (Structure / Graph / Design)

Результат извлечения — это не единый текст, а три независимых, тематически разделённых поля, которые в коде представлены структурой `FBPR_ExtractedData` (`Structure`, `Graph`, `Design` — все типа `FText`, плюс `AssetType` и `AssetName`). Разделение на три поля продиктовано и удобством чтения человеком, и удобством работы AI-ассистента: LLM гораздо эффективнее работает с текстом, разбитым на тематические блоки, чем с одной сплошной простынёй.

- **Structure** — «статический» срез Blueprint: переменные (с типами и, где применимо, значениями по умолчанию), функции и их сигнатуры, компоненты (для Actor/ActorComponent), реализуемые интерфейсы, exposed-свойства и т.п. Это то, что отвечает на вопрос «из чего это состоит».
- **Graph** — «динамический» срез: содержимое графов событий и функций, цепочки нод, связи по exec-пинам и по данным, привязки обработчиков к событиям (в том числе, для виджетов, привязки вида `K2Node_ComponentBoundEvent` между визуальным элементом и обработчиком в коде). Это то, что отвечает на вопрос «что это делает».
- **Design** — специфичный для виджетов срез: иерархия UMG-виджетов, их взаимное вложение, ключевые визуальные и раскладочные свойства (например, оверрайды `SizeBox`, видимость, привязки текста и т.п.). Заполняется только для типов ассетов, у которых есть визуальная составляющая; для остальных типов поле остаётся равным сентинел-значению «N/A» или «No Data found» (см. ниже).

Каждое из трёх полей по умолчанию форматируется в Markdown (заголовки, списки, выделения) согласно внутреннему флагу `EOutputFormat` (значение по умолчанию — `Minimal`, наиболее компактный вариант, оптимизированный под экономию токенов при вставке в LLM-чат) — это отдельный, внутренний уровень форматирования кода, не путать с пользовательской настройкой формата экспортируемого файла (`EBPR_ExportFormat`, см. раздел «Настройки плагина» ниже), которая относится уже к тому, в каком расширении файл сохраняется на диск.

Отдельно стоит подчеркнуть смысл «пустых» значений. Если у конкретного типа ассета логически нет данных для той или иной вкладки (например, Design для Actor Blueprint, или Graph для User-Defined Structure), поле не остаётся физически пустой строкой, а получает одно из предсказуемых заполнительных значений — `"N/A"` или `"No Data found"`. Это не косметическая деталь: именно по этим двум сентинел-строкам логика экспорта (`IsMeaningful()`, см. ниже) отличает «здесь реально нечего показывать» от «здесь есть контент», и не включает пустые разделы в итоговый экспортируемый документ.

Наконец, у каждого извлечённого результата есть сопутствующий `AssetName` (имя исходного ассета) — оно не выводится как отдельная «вкладка», но используется как имя файла по умолчанию при экспорте, обеспечивая, что сохранённый файл сразу узнаваем по названию, даже если пользователь извлекал подряд данные нескольких разных Blueprint.

## Настройки плагина

Настройки живут в стандартном разделе редактора **Project Settings → Plugins → Blueprint Reader** (класс `UBPR_Settings`, наследник `UDeveloperSettings`, конфигурация уровня редактора — `config = Editor`). Выбор `UDeveloperSettings` — намеренное архитектурное решение: это штатный, знакомый любому UE-разработчику способ конфигурировать плагин, не требующий отдельного окна настроек внутри самого BlueprintReader, и автоматически интегрирующийся в поиск по Project Settings.

Доступны три настройки, сгруппированные в две категории:

**Категория «Output»:**
- **Export Format** (`ExportFormat`, перечисление `EBPR_ExportFormat`) — определяет формат файла при экспорте на диск. Два варианта: **Markdown (.md)** (значение по умолчанию) и **Plain Text (.txt)**. Эта настройка напрямую управляет расширением файла, предлагаемым фильтром в диалоге сохранения, и, соответственно, ожиданиями о том, будет ли в сохранённом тексте разметка Markdown или чистый текст без нее.

**Категория «Widget»:**
- **Restrict Widget Recursion Depth** (`bRestrictWidgetRecursionDepth`, булево, по умолчанию `true`) — включает ограничение глубины обхода дерева вложенных виджетов при извлечении. Виджеты в UMG нередко вкладываются друг в друга на много уровней (панель → контейнер → под-виджет → под-под-виджет...), и без ограничения глубины обход рекурсивного дерева на насыщенном UI-ассете рискует сгенерировать чрезмерно длинный вывод, малопригодный для вставки в чат с LLM (лимиты контекста, потеря читаемости).
- **Widget Recursion Depth** (`WidgetRecursionDepth`, целое число, диапазон 1–50, по умолчанию 10) — собственно значение глубины, до которой доходит рекурсивный обход (`ProcessWidgetHierarchy`). Поле активно и редактируемо только тогда, когда включен `bRestrictWidgetRecursionDepth` (`EditCondition` в UPROPERTY) — то есть в UI настроек оно визуально блокируется, если ограничение выключено, чтобы не создавать ложного впечатления, будто значение на что-то влияет само по себе.

С точки зрения пользовательского сценария эти две настройки существуют в первую очередь для баланса между полнотой извлечения и практичностью результата: пользователи, работающие с очень глубоко вложенными пользовательскими виджетами (например, сложные HUD с множеством переиспользуемых `WBP_*`-компонентов), могут либо увеличить глубину, чтобы получить более полную картину, либо отключить ограничение полностью, если полнота важнее компактности. Значение по умолчанию (10 уровней, ограничение включено) было выбрано как разумный компромежуточный уровень, покрывающий подавляющее большинство реальных UI-иерархий без риска получить лавинообразно длинный вывод.

Все три настройки читаются из `UBPR_Settings` через `GetDefault<UBPR_Settings>()` в момент извлечения (для глубины рекурсии — внутри `BPR_Extractor_Widget`) и в момент экспорта (для формата файла — внутри `SBPR_TabSwitcher`), то есть изменение настроек в Project Settings применяется к следующему извлечению/экспорту сразу, без необходимости перезапускать редактор.

## Экспорт данных

Кнопка **«Export to file…»** в футере окна вывода превращает уже извлечённые данные (`Structure`, `Graph`, `Design` из текущей вкладки-сессии) в единый текстовый документ и сохраняет его на диск через нативный диалог сохранения файла операционной системы.

**Сборка документа.** Итоговый файл — это конкатенация трёх полей в фиксированном порядке **Structure → Graph → Design**, разделённых пустыми строками, но с важной оговоркой: раздел включается в документ, только если он содержательный. Проверка содержательности (внутренняя функция-предикат) обрезает пробелы по краям текста и отбрасывает раздел, если после этого он оказывается пустым, либо равен одному из двух сентинел-значений — `"N/A"` или `"No Data found"`. Иными словами, если пользователь экспортирует, скажем, User-Defined Structure (у которой есть только Structure, а Graph и Design — пустышки), в файл попадёт исключительно содержимое Structure, без двух строк-заглушек, засоряющих документ. Итоговый текст дополнительно обрезается от завершающих пробельных символов.

**Сценарий «нечего экспортировать».** Если после сборки документ оказывается полностью пустым (то есть все три раздела оказались незначимыми), кнопка не открывает диалог сохранения вообще, а сразу показывает уведомление **«Nothing to export for this asset.»** с иконкой неудачи — пользователь не тратит время на выбор пути сохранения ради пустого файла.

**Выбор формата и диалог сохранения.** Формат берётся из настройки `ExportFormat` (см. выше): `Markdown` даёт расширение `.md` и фильтр «Markdown Document (*.md)|*.md», любое другое значение (либо отсутствие объекта настроек) — расширение `.txt` и фильтр «Text Document (*.txt)|*.txt» (то есть Markdown — поведение по умолчанию и в отсутствие явно заданных настроек). Имя файла по умолчанию формируется как `{AssetName текущих данных, либо "BlueprintExport", если оно не задано}.{расширение}`. Диалог сохранения — это нативный, платформенный `IDesktopPlatform::SaveFileDialog`, открываемый в директории `FPaths::ProjectSavedDir()` проекта, с заголовком «Export Blueprint Data»; родительское окно для диалога определяется через `FSlateApplication::FindBestParentWindowHandleForDialogs`, чтобы диалог модально привязывался к окну редактора, а не «терялся» за другими окнами.

**Отмена и результат.** Если пользователь закрывает диалог без выбора файла — плагин молча завершает операцию, без каких-либо уведомлений (это ожидаемое, ненавязчивое поведение: отмена сохранения — не ошибка). Если файл выбран, документ записывается через `FFileHelper::SaveStringToFile` в кодировке **UTF-8 без BOM** — важная деталь для корректного отображения кириллицы и любых не-ASCII символов (например, русскоязычных комментариев в Blueprint) в большинстве текстовых редакторов и Markdown-вьюеров. Об успехе или неудаче записи пользователь узнаёт из всплывающего редакторского уведомления (`FSlateNotificationManager`) с временем жизни 4 секунды и соответствующей иконкой успеха/неудачи: **«Exported to {имя файла}»** при успехе (плюс запись в лог) либо **«Export failed (could not write file).»** при неудаче (плюс предупреждение в лог) — например, если целевой путь оказался недоступен для записи.

Таким образом, экспорт спроектирован как максимально «тихая», предсказуемая операция: минимум диалогов, чёткая обратная связь только тогда, когда она действительно нужна (успех/неудача записи, отсутствие данных), и никакого шума при штатной отмене.

## Пользовательские сценарии (флоу использования)

**Сценарий 1 — «Объясни этот Blueprint».** Разработчик наследует чужой (или давно не трогаемый собственный) Actor Blueprint и не понимает, что происходит в его графе событий. Он кликает правой кнопкой по ассету → «Read Blueprint for AI-Assistant» → в открывшемся окне читает вкладку Structure (какие переменные и компоненты есть) и Graph (какая логика связывает события), при необходимости нажимает «Export to file…», сохраняет `.md`-файл и вставляет его содержимое в чат с AI-ассистентом с вопросом «объясни, что делает этот Blueprint». Поскольку вывод уже структурирован по разделам и отформатирован в Markdown, LLM не тратит контекст на разбор визуального шума и сразу даёт содержательный ответ.

**Сценарий 2 — «Помоги с багом в виджете».** UI-программист видит, что вложенный `WBP_*`-виджет на экране отображает не то значение, которое ожидалось. Он открывает виджет через BlueprintReader, попадает сразу на вкладку **Design** (виджеты открываются с неё по умолчанию), видит иерархию вложенности и exposed-свойства конкретного экземпляра (включая нестандартные значения, установленные именно на этом экземпляре — например, `In Text: Clan`), затем переключается на вкладку **Graph**, чтобы проверить привязки событий к обработчикам. Комбинация Design + Graph даёт AI-ассистенту полную картину и «что видно», и «что исполняется» — без необходимости пользователю самому пересказывать структуру словами.

**Сценарий 3 — «Ассет не поддерживается».** Пользователь по ошибке (или из любопытства) пытается прочитать, например, Animation Blueprint или Data Table. После клика по пункту меню вместо окна с данными появляется компактное окно-предупреждение с понятным сообщением о том, что тип ассета пока не поддерживается, и ссылкой на GitHub-репозиторий для проверки обновлений. Пользователь не получает пустого или сломанного вывода — сообщение однозначно и не оставляет впечатления, что что-то работает неправильно.

**Сценарий 4 — «Быстрая документация для команды».** Тимлид перед код-ревью или онбордингом нового сотрудника хочет иметь текстовое описание ключевых Blueprint-классов проекта. Он последовательно прогоняет через BlueprintReader несколько Actor- и Component-Blueprint'ов, каждый раз нажимая «Export to file…» и сохраняя результат в папку с документацией проекта (`Saved/` или отдельную папку docs) — без необходимости вручную набирать описание структуры, разработчик получает актуальный на момент экспорта текстовый снимок.

**Сценарий 5 — «Настройка глубины для сложного UI».** Дизайнер интерфейсов работает над насыщенным HUD с множеством вложенных переиспользуемых виджетов и замечает, что вывод BlueprintReader для корневого экрана обрывается раньше, чем ожидалось (глубина по умолчанию — 10 уровней). Он заходит в Project Settings → Plugins → Blueprint Reader, увеличивает `Widget Recursion Depth` либо отключает `Restrict Widget Recursion Depth` полностью, и повторно открывает тот же ассет через контекстное меню — новое значение настройки применяется сразу, без перезапуска редактора.

## Ограничения текущей версии

Текущая версия BlueprintReader (0.2.0, статус беты — `IsBetaVersion: true`) сознательно ограничена по охвату типов ассетов и по глубине некоторых сценариев. Ниже — полный перечень известных ограничений, которые пользователь должен понимать до начала работы:

- **Ограниченный набор поддерживаемых типов.** Поддерживаются только восемь категорий ассетов (Actor, ActorComponent и Widget Blueprint'ы, Blueprint-интерфейсы, перечисления и структуры пользователя, материалы и материальные функции — см. раздел «Поддерживаемые типы ассетов»). В частности, **Animation Blueprints не поддерживаются**, несмотря на то что формально являются `UBlueprint`-производным классом — это частый источник путаницы у новых пользователей. Также не поддерживаются StateTree/Behavior Tree ассеты, Niagara-системы, Data Assets/Data Tables и любые обычные Blueprint-классы, не являющиеся ни Actor, ни ActorComponent, ни интерфейсом.
- **Отсутствие фильтрации по подграфу.** Плагин извлекает граф целиком; выборочный экспорт «только эта функция» или «только этот участок графа» (запланированная возможность частичной фильтрации графа) в текущей версии отсутствует и отложен на будущее.
- **Отсутствие индикатора прогресса.** Извлечение данных для крупных, насыщенных ассетов выполняется синхронно, без визуального индикатора хода выполнения; на практике операция достаточно быстрая для подавляющего большинства ассетов, но в UI нет обратной связи о том, что именно происходит во время обработки особенно тяжёлых графов.
- **Разрешение изображений/кистей в Design-выводе.** Для виджетов, использующих `Brush`/`Image` со ссылкой на внешний ресурс (`ResourceObject`), вкладка Design в текущей версии выводит сырой идентификатор (GUID) ресурса вместо человекочитаемого имени ассета — известный пункт улучшения качества вывода, ещё не реализованный.
- **Незначимые цветовые значения не отфильтрованы.** Design-вывод может содержать визуально бесполезные строки вида `BackgroundColor:(0,0,0,0)` (полностью прозрачный/нулевой цвет) — фильтрация таких незначимых значений запланирована, но пока не реализована.
- **Единая ветка исходного кода для двух версий движка (5.7 и 5.8).** Плагин собирается и проверен на обеих версиях движка из одного исходного дерева без версионных `#if`-развилок (файл `BPR_Compat.h` зарезервирован под будущие точечные исправления, но на момент написания документа пуст). Риск в том, что будущие версии движка потенциально потребуют версионно-специфичных правок — этот механизм подготовлен, но конкретных обходных путей пока нет, поскольку не в чем.
- **Отсутствие автоматизированного CI и формальных смок-тестов.** Совместимость с 5.7/5.8 подтверждена ручной сборкой и ручным сравнением вывода на реальных ассетах, но автоматизированный компиляционный конвейер (CI compile-matrix) и формализованные смок-тесты пока не выстроены — это входит в отдельный, ещё не начатый этап работ (следующий milestone продукта).
- **Публикация на Fab ещё не завершена по обеим версиям движка.** Поскольку один `.uplugin` не может одновременно объявлять поддержку нескольких версий движка для публикации на Fab (нужны отдельные «Project Version» с проштампованным `EngineVersion` для каждой), процесс подготовки раздельных пакетов под 5.7 и 5.8 для маркетплейса на момент написания документа ещё не выполнен.
- **Статус беты.** Плагин помечен как `IsBetaVersion: true` — это отражает то, что этап контроля качества перед стабильным релизом (автоматизированное тестирование, CI, полноценная публикация на Fab) ещё не пройден, и по SemVer-конвенции версия сознательно не поднята до 1.0.0.
