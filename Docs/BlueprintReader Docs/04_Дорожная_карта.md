# Дорожная карта — BlueprintReader

> Нумерация вех (`M1`–`M6`) соответствует истории разработки плагина и совпадает с обозначениями, используемыми в техническом журнале проекта (`CLAUDE.md`). Статусы актуальны на момент версии `0.2.0` (beta).

## Пройденные вехи

**M1 — Фабрика экстракторов и стабильная сборка.** *Завершено.*
Заложена базовая архитектура плагина: `BPR_Core` выступает фабрикой, которая владеет и регистрирует все экстракторы (`RegisterAllExtractors`, `IsSupportedAsset`, `ExtractAsset`, `GetUnsupportedAssetInfo`), а `BPR_Types.h` определяет единый контракт данных (`FBPR_ExtractedData` со полями `Structure`, `Graph`, `Design`). Установлена иерархия базовых классов: `BPR_Extractor_Base` → `BPR_Extractor_Object` → `Actor` / `ActorComponent` / `Widget`, а также независимые ветки `Interface`, `Enum` / `Structure`, `Material`, `MaterialFunction`. По итогам вехи сборка плагина приведена в чистое состояние на новой структуре.

**M2 — Widget как полноценный наследник `BPR_Extractor_Object`.** *Завершено, валидировано в редакторе.*
Экстрактор виджетов был переведён с автономной реализации на общую иерархию, что устранило дублирование логики и заметно сократило объём кода:

- Виджет теперь наследует `BPR_Extractor_Object` вместо отдельного пути извлечения.
- Удалён устаревший переходник `WidgetAdapter`.
- Убрана дублирующая логика извлечения графов и переменных, общая с базовым классом (~730 строк кода).
- Привязки событие → обработчик (`AppendWidgetEventBindings` / `K2Node_ComponentBoundEvent`) переведены на единый механизм; удалён неиспользуемый старый `AppendWidgetBindings`.
- Логирование виджетов переведено на унаследованную базовую категорию `LogBlueprintReader` вместо разрозненного логирования.
- Отдельно решена проблема извлечения вложенных пользовательских виджетов (`WBP_*`, `UUserWidget`): у вложенных экземпляров `WidgetTree` на уровне ассета равен `null`, из-за чего обычная рекурсия по дереву (`ProcessWidgetHierarchy`) не срабатывала. Решение ("Approach A" в `HandleUnknownWidget`) — перебор собственных экспонированных пользовательских переменных виджета (`IsUserVariable`) и вывод их нестандартных значений экземпляра (`GetPropertyDefaultValue`) в отдельный блок "Exposed Properties (set on this instance)". Валидировано вручную: вложенные текстовые метки и параметры (например, `In Text: Clan`, `MaxCharToEnter`) теперь корректно отображают реальные значения конкретного экземпляра.

Итог вехи: `BPR_Extractor_Widget.cpp` сократился примерно с 3445 до 2660 строк. Все сборки проверены на UE 5.7.

**M3 — Material / MaterialFunction / ActorComponent в иерархии экстракторов.** *Завершено, валидировано в редакторе, влито в `main`.*
Три экстрактора, ранее существовавшие обособленно, встроены в общую иерархию с корректными приоритетами: `Material` → `BPR_Extractor_Base` (приоритет 40), `MaterialFunction` → `BPR_Extractor_Base` (приоритет 30), `ActorComponent` → `BPR_Extractor_Object` (приоритет 90). Каждый экстрактор реализует полный контракт (`Process` / `CanHandleAsset` / `GetPriority`) и — что важно — теперь корректно устанавливает `OutData.AssetType`, тогда как ранее это поле оставалось незаполненным для всех трёх типов (системная ошибка, из-за которой интерфейс считал эти ассеты неподдерживаемыми). Экстрактор `ActorComponent` избавлен от дублирующего стека извлечения Blueprint-графов, за счёт чего сократился с 733 до 76 строк — по аналогии с рефакторингом M2.2. Обход DAG выражений материала (Expression DAG) сознательно оставлен без изменений и не унифицирован с `Object`-веткой — это самый рискованный участок кода, и он отдельно помечен для проверки при будущей работе с версиями движка. Опциональная "Фаза 3" (обобщённый экстрактор-заглушка `BPR_Extractor_Blueprint` для произвольных Blueprint-классов без более специфичного совпадения) признана необязательной и не входила в объём M3.

**M4 — UX: настройки плагина и экспорт в файл.** *Завершено, валидировано в редакторе, влито в `main`.*
Веха ограничена двумя подпунктами (4.1 и 4.2); один пункт исключён из плана, два — сознательно отложены (см. раздел "Отложенные и опциональные задачи").

- **4.1 — Настройки плагина.** Добавлен `UBPR_Settings : UDeveloperSettings` (`Public/Core/BPR_Settings.h`, `config=Editor`), доступный в Project Settings → Plugins → Blueprint Reader. Экспонирует формат экспорта (`EBPR_ExportFormat`: Markdown/PlainText), флаг ограничения глубины рекурсии по дереву виджетов (`bRestrictWidgetRecursionDepth`) и саму глубину (`WidgetRecursionDepth`, диапазон 1–50). `ProcessWidget` теперь читает эти настройки через `SetRecursionSettings()` вместо ранее захардкоженного значения `MaxDepth=10`. Технически значим факт: это первый `UObject` в модуле, из-за чего для плагина впервые заработал UnrealHeaderTool (UHT).
- **4.2 — Экспорт в файл.** В `FBPR_ExtractedData` добавлено поле `FString AssetName`, централизованно заполняемое в `BPR_Core::ExtractAsset`. В подвале `SBPR_TabSwitcher` (переживает перестроение вкладок) добавлена кнопка "Export to file…", вызывающая `IDesktopPlatform::SaveFileDialog` (имя файла по умолчанию — имя ассета, расширение — из настроек формата экспорта), после чего данные записываются через `FFileHelper` в UTF-8 с уведомлением об успехе или ошибке. По ходу реализации устранена версионная особенность UE 5.7: `FindBestParentWindowHandleForDialogs` возвращает `const void*`.

**M6 — Совместимость с UE 5.7 и 5.8.** *Техническая часть завершена (см. подробности в разделе "Текущая веха" — процессные хвосты вехи ещё открыты).*

## Текущая веха

Техническая работа по **M6 (совместимость с UE 5.7 и 5.8)** на текущий момент фактически завершена, однако формально веха остаётся открытой до закрытия сопутствующих процессных задач, часть которых пересекается с M5. Ниже — детальный статус.

**Подход.** Выбрана стратегия "A+C": единое дерево исходного кода (не отдельные ветки под каждую версию движка — такой вариант отклонён из-за риска расхождения кода) с версионными развилками через `UE_VERSION_OLDER_THAN`, централизованными в заголовке `BPR_Compat.h`. Гейтом надёжности служит CI-матрица сборки под 5.7 и 5.8 одновременно (общая с задачей M5.2).

**Результат Фазы 1.** Плагин собирается без ошибок и на UE 5.7, и на UE 5.8; при этом `BPR_Compat.h` остаётся **пустым** — ни одной версийной развилки (`#if`) в итоге не потребовалось. Первая попытка сборки через `RunUAT BuildPlugin` под 5.8 упёрлась не в проблему кода, а в артефакт тулчейна (длина пути превысила 260 символов, сработал `CheckPathLengths`) — устранено использованием более короткого пути упаковки. Единственная реальная находка, потребовавшая правки кода: заголовок `Engine/UserDefinedStruct.h` (использовался в `BPR_Extractor_Structure.h`/`.cpp`) в 5.8 убран как переходник — исправлено бесплатно, без версионной развилки, переключением на канонический путь `StructUtils/UserDefinedStruct.h` (модуль CoreUObject), который не изменился в обеих версиях движка. Показательно, что обход DAG выражений материала — участок кода, заранее помеченный как самый рискованный, — собрался на 5.8 без единой правки, только с шумовыми предупреждениями заголовков движка. Повторная проверка подтвердила: сборка под 5.7 после этой правки по-прежнему без ошибок, регрессий не внесено.

**Задача 6.0 — два подтверждённых бесплатных исправления (без `#if`):**

- **(a)** Прямое чтение полей `USizeBox::bOverride_*` в `BPR_Extractor_Widget.cpp` (`HandleSizeBoxProperties`, ~строки 2068–2084) заменено на рефлексию через `FindFProperty<FBoolProperty>`, по образцу уже существующего паттерна `AppendBoolFlag`, используемого для ListView/TreeView в том же файле (~строка 2263).
- **(b)** Из `BlueprintReader.Build.cs` удалены неиспользуемые зависимости `StateTreeModule` / `GameplayStateTreeModule` / `StateTreeEditorModule` — рудиментарные модули, создававшие риск ошибки компоновки на хосте с UE 5.8 без включённого плагина StateTree.

Статус на данный момент: оба пункта подтверждены как выполненные и присутствующие в исходном коде.

**Задача 6.3 — runtime smoke-diff.** Выполнена вручную: пользователь сравнил вывод плагина на реальных ассетах между редакторами 5.7 и 5.8 — результаты совпадают, расхождений в извлечённых данных не обнаружено.

**Эволюция окружения разработки.** Изначально поддерживались живой хост на 5.7 (`PluginsProject`) и его замороженная резервная копия (`PluginsProject57`). После завершения технической части M6 основной хост `PluginsProject` пересоздан заново на UE 5.8 и подтверждённо собирается полным редакторским модулем без ошибок; `PluginsProject57` при этом сохранён как хост-справочник для отката на 5.7 по требованию (обычный git-чекаут того же репозитория) — также подтверждённо собирается без ошибок. Таким образом, текущая топология: `PluginsProject` — основной хост разработки на 5.8, `PluginsProject57` — резервный хост для проверки совместимости с 5.7.

**Что остаётся в рамках M6 (процессные, не технические, задачи):**

- Перепроверить фактическое применение пункта 6.0(a).
- Развернуть CI-матрицу сборки 5.7+5.8 как автоматизированный пайплайн (общая веха с M5).
- Подготовить упаковку под Fab для каждой версии движка отдельно (см. ниже, пересекается с M5).

## Планируемые вехи

**M5 — CI, smoke-тесты, релиз v0.2 на Fab.** *Не начато как отдельная работа; часть смежных задач уже закрыта в рамках M6.*

Как самостоятельный фронт работ M5 ещё не начат, но часть его логических предпосылок уже выполнена попутно при работе над M6 (в частности, ручной runtime smoke-diff по задаче 6.3). Что остаётся открытым непосредственно для M5:

- **CI-матрица сборки (5.7 + 5.8).** Пока сборка под обе версии движка проверяется только вручную локально; требуется автоматизированный пайплайн, который будет служить общим гейтом надёжности с M6.
- **Формальные smoke-тесты.** Ручная проверка (M6.3) была разовой и не оформлена в виде повторяемого тестового набора — требуется формализация.
- **Подготовка и упаковка релиза v0.2 для Fab**, включая механизм упаковки под каждую версию движка отдельно: поле `EngineVersion` в `.uplugin` может содержать только одну версию, поэтому реальный механизм Fab — отдельная загружаемая "Project Version" на каждую поддерживаемую версию движка, каждая со своей копией `.uplugin`, в которой `EngineVersion` проставлен именно под неё (5.7.0 / 5.8.0) непосредственно на этапе упаковки. Это отдельная фаза плана миграции (Фаза 4), которая пока не выполнена.

Версия плагина сознательно не поднята до `1.0.0` — этот шаг зарезервирован за моментом завершения M5 (CI, smoke-тесты, релизное QA); до тех пор в манифесте плагина сохраняется `IsBetaVersion: true`.

## Отложенные и опциональные задачи

Ниже перечислены задачи, по которым принято осознанное решение отложить или не включать их в объём завершённых вех. Это не забытые пункты, а зафиксированный бэклог.

- **M4.3 — Фильтр частичного графа.** *Отложено.* Возможность выборочно ограничивать объём извлекаемого графа (например, только определённые функции/события) не вошла в объём M4 и перенесена на будущее без привязки к конкретной вехе.
- **M4.4 — Индикатор прогресса.** *Отложено.* UI-индикатор процесса извлечения для крупных ассетов не реализован в M4; остаётся открытым пунктом бэклога.
- **M4.5 — Ссылки в README (Patreon/Boosty).** *Исключено из плана (dropped), не просто отложено.* Планы по монетизации через Patreon/Boosty приостановлены; соответствующие ссылки не добавлялись и не планируются в текущем горизонте.
- **Фаза 3 экстракторов (M3) — обобщённый `BPR_Extractor_Blueprint`.** *Опционально, не запланировано.* Универсальный экстрактор-заглушка для произвольных Blueprint-классов, не подпадающих ни под один специализированный тип (Actor/ActorComponent/Widget/Interface), признан необязательным улучшением, а не требованием текущей архитектуры.
- **Прочие открытые улучшения качества вывода (без привязки к конкретной вехе):**
  - Дедупликация `CleanName` на уровне базового класса (на сегодня остался только локальный дубликат в Widget-экстракторе).
  - Повсеместный переход с `LogTemp` на собственную категорию `LogBlueprintReader` в файлах вне Widget-экстрактора (UI, Material, ActorComponent).
  - В выводе вкладки Design — разрешение `ResourceObject` кистей/изображений в человекочитаемое имя ассета вместо необработанного GUID.
  - Подавление в выводе Design полностью нулевых/прозрачных цветовых значений (например, `BackgroundColor:(0,0,0,0)`), которые не несут полезной информации для LLM.

Эти пункты не блокируют релиз v0.2 и рассматриваются как материал для последующих итераций после стабилизации M5/M6.
